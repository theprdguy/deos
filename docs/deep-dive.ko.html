이 OS가 실제로 작동하는 법

README가 요약한 부분을 한 단계 더 — 레이어·에이전트·강제 장치까지. "안전하다"가 아니라 왜 안전한지 이해하고 싶은 사람을 위한 글.

0한 문단

Vibe Coding OS는 "AI에게 코드를 시키는 일"을 감사 가능한 상태 기계로 바꿉니다. 모든 진실은 devos/ 아래 파일에 있고, 일은 티켓으로만 움직이며, 누가 무엇을 왜 했는지는 티켓별 전이 이력과 세션 로그에 남습니다.

사용자(PM)가 의도를 말한다 → CLAUDE1이 티켓으로 분해하고 빠진 에러·엣지·보안 케이스를 캐묻는다 → 사용자가 plan을 승인한다 → 티켓 owner(builder 세션 내 / CODEX 외부)가 구현한다 → 베이스라인 게이트 + 읽기 전용 리뷰 보드가 검토한다 → 위험·불확실하면 다른 벤더(CODEX b')가 2차 의견 → BLOCKER 없으면 done, 있으면 blocked + 결정 큐.

13층 구조

세 겹입니다. 위 두 겹은 "무엇을 한다", 맨 아래가 "그걸 못 어기게 한다". 안전성의 대부분이 3층에서 옵니다.

층구성역할
1 · 진실파일 기반 SSOT (devos/)프로젝트가 무엇이고 어디까지 왔는지 기억
2 · 행위자계획가 + 구현가 2 + 읽기전용 리뷰 보드계획·구현·검토
3 · 강제도구 권한 · 훅 · 게이트 · grep · 잠긴 결정규칙을 "신뢰"가 아니라 "구조"로 막음

핵심 원리: 역할·권한이 분리되고, 권한은 도구 차원에서 제한된다. 검토자는 Edit/Write 툴이 없어 "고치고 싶어도 못 고친다" — 자기검토 객관성 붕괴가 원천 차단.

2에이전트

각자 다른 모델·다른 도구를 씁니다. 권한 분리는 선의가 아니라 구조로 강제됩니다.

CLAUDE1 (main)Opus계획·지휘

사용자와 대화. PRD를 티켓으로 분해·조사·라우팅. 코드를 직접 안 쓰고, 결과를 직접 OK 하지 않음 — 리뷰 보드로 넘김.

builderSonnetRead·Edit·Write·Bash

세션 내 구현가. 모호·감성 중요한 UI, 새 UX 흐름에 우선. 티켓 files 범위만 수정.

CODEX외부 CLIsubprocess·sandbox

플랫폼 구현가 — 백엔드/API/데이터/인프라/테스트/마이그레이션 — + 교차모델 2차 의견(b'). 다른 벤더라 Claude의 공통 맹점을 잡음.

reviewer · designer · securityREAD-ONLY

리뷰 보드(Opus/Sonnet). reviewer=적대적 PR 검토(DOD↔테스트·단언·범위·계약). designer=UI/UX(일관성·상태·접근성). security=OWASP+STRIDE, auth/결제/권한/외부입력 자동 발동. 셋 다 Edit/Write 툴 없음.

읽기전용이 핵심인 이유

대부분 팀은 "자기 PR 자기 머지 금지"를 정책으로 둡니다. 여기는 권한입니다 — 검토자에게 write 툴이 없어 발견만 하고 조용히 못 고칩니다. 구조적 객관성.

3티켓 — 일의 단위

모든 구현은 devos/tasks/QUEUE.yaml의 티켓입니다. 1 티켓 = 1 PR. 자기완결적 — 구현가가 티켓 + SSOT만 읽고 실행 가능해야 합니다.

- id: T-AUTH-LOGIN-01
  status: todo            # 디스패처는 todo만 집음
  owner: CODEX            # BUILDER | CODEX
  mode: production        # 게이트 자세 (§6)
  tdd: required
  security_audit: true    # auth/결제/권한 자동 강제
  goal: |   POST /auth/login
  dod:
    - 'valid 자격증명 -> 200 + access/refresh JWT'     # 성공
    - 'wrong password -> 401 + "invalid credentials"'  # 에러(필수)
  files: [apps/api/src/auth/login.py, tests/test_login.py]
  verify: |  python3 -m pytest tests/test_login.py -v
  gates: [pr-check]

DOD 규칙: 성공 DOD가 있으면 에러 DOD 필수. 비개발자가 happy path만 적는 빈틈을 시스템이 메우는 장치.

상태 머신 + 감사 추적

상태: todo doing code_ready needs_pm done blocked parked. 전이마다 actor·reason·timestamp가 _transition_history에 누적 — 티켓이 왜 이 상태인지 채팅이 아니라 파일에서 재구성.

5디스패치 (결정적)

단계동작
1 · lookup+preflightstatus==todo? deps 모두 done? files 존재?
2 · owner 라우팅BUILDER → 세션 내 / CODEX → subprocess
3 · 구현구현가가 자기 컨텍스트에서 작업
4 · pr-check시크릿 · 계약 동기화 · 범위 가드 · 세션 로그 · TDD 첫커밋
5 · 리뷰 체인티켓 분류 → 필요한 읽기전용 검토자 spawn (+5% 무작위 표본은 가벼운 경로도 승격)
6 · fan-in + b'verdict 취합; 정량 조건 충족 시 CODEX 2차 의견
7 · 최종BLOCKER 우선 판정 → 상태 + 로그 + 표준 헤더
결정적 엔진 + 규율적 래퍼

엔진(dispatcher.py)은 코드로 결정적 — singleton 락, HEAD-SHA diff 기준선(git stash 안 씀), 게이트, 1회 리트라이. 그 위 오케스트레이션(분류·b' 트리거)은 CLAUDE1이 위 프로토콜대로 실행.

6동작 모드

하나의 아이디어가 신뢰도가 오르며 exploration → productization → production으로 이동. 모드가 게이트 자세를 정하고 디스패처가 강제합니다.

게이트explorationproductizationproduction
Tests선택/report-only전략 필수필수
Reviewerreport-only권장필수 · reject 차단
Security시크릿/auth만리스크 식별트리거 시 필수
성공+실패 DOD선택필수필수

Report-only는 fail-closed: 모드가 exploration/productization 이고 명시적 soft 게이트일 때만 완화. 시크릿·범위 위반·destructive·미인식 게이트는 모든 모드에서 차단. Production 예외는 기록된 웨이버(승인자·위험·follow-up) 필요.

7안전 — 신뢰가 아니라 구조

🔒 권한으로 강제된 객관성

검토자는 읽기전용 도구. 계획가는 훅이 apps/ packages/ scripts/ tests/ 쓰기를 차단. 둘 다 약속이 아니라 도구 차원의 사실.

⚖️ Iron Laws (전 에이전트)

실패 테스트 없이 prod 코드 없다 · 근본원인 없이 수정 없다 · 새 증거 없이 "완료" 없다 · 적대적 검토 없이 머지 없다 · 범위축소 어휘 없다(grep 강제).

🚪 항상 켜진 안전 바닥

시크릿 노출·owner mismatch·범위 위반·미해결 deps·dirty-tree destructive는 모드 무관 차단, 일반 웨이버로 우회 불가.

🤝 교차모델 안전망(b')

검토자 불확실·BLOCKER·고위험 티켓이면 다른 벤더가 2차 의견. 평소엔 비용 0.

사고 → 잠긴 결정

무언가 부서지면 그 수정은 강제 규칙이 됩니다 — 위반이 자동 reviewer BLOCKER인 D-XX Locked Decision. 그래서 재발 못 함. 규칙집은 위시리스트가 아니라 과거 사고의 기록.

8LLM이 OS를 읽는 법 (SSOT)

모든 상태가 파일. 세션 시작 시 다시 읽으므로 공유 채팅창 드리프트가 없습니다. 충돌 시 위가 강함:

1. PROJECT_STATE.md        마일스톤·현재 동작
2. API_CONTRACT / UI_CONTRACT   인터페이스 계약 (코드보다 우선)
3. ADR/*                   아키텍처 결정
4. tasks/QUEUE.yaml        활성 티켓
5. 코드
6. 세션 로그
7. 채팅 로그               (가장 안 믿음)

Contract-first: API/UI 동작 변경 시 계약 문서를 먼저 고침; 불일치는 reviewer가 BLOCKER.

9host-OS 모델

엔진·에이전트·도리는 호스트 레포에 한 번만 존재. 각 제품은 projects/ 아래 독립 git 레포로, 자기 코드와 task 상태만 보유.

~/dev-os/            호스트 = 엔진 · 도리 · 공유 .claude (git 1개)
|-- server/  bin/os3  .claude/  devos/
`-- projects/        (호스트가 git-ignore — 각자 독립 레포)
    |-- product-a/   apps/ packages/ + 자기 devos/tasks · PROJECT_STATE · .os3.yaml
    `-- product-b/   ...
  • 엔진은 1곳 — 고치면 전 프로젝트 적용. sync 없음.
  • 프로젝트는 독립 — 자기 레포·상태, 따로 배포.
  • 진입은 하나 — os3 open <name>이 호스트 설정을 달고 진입.

단일 프로젝트로 써도 됨 — devos/를 상태로 쓰고 projects/ 무시. host-OS는 여러 제품으로 확장하는 방식.

10OS-feedback 루프

OS는 자기를 demand-pulled로 개선합니다: 제품 작업 중 마찰을 os3 feedback "..."로 devos/os-feedback/INBOX.md에 포착하고, 세션 시작 시 검토하고, 주기적 consolidation 패스에서 티켓으로 전환.

왜 load-bearing인가

OS 개선이 "제품 마찰에서" 온다면 그 마찰을 포착해야 합니다 — 안 그러면 채팅에서 한 번 말하고 증발. 반응적 수정은 부딪힌 것만 잡고, 주기적 consolidation은 안 밟는 잠재 부채를 잡습니다.

12엔진 내부 — 락 · 스냅샷 · 리트라이

§5가 "디스패치가 무슨 단계를 밟나"였다면, 여기는 엔진이 그 단계를 어떻게 안전하게 실행하나입니다. 핵심 셋: 동시 실행 방지 · 변경 기준선 고정 · 실패 시 1회 리트라이. 빠진 것에 주목 — git을 망가뜨리는 rollback이 없습니다. 복구는 설계상 읽기 전용입니다.

Singleton 락 — 프로젝트당 디스패처 둘 금지

디스패치는 ~/.os3/dispatcher.<project>.pid에 원자적 파일 생성(O_CREAT|O_EXCL)으로 락을 잡습니다. 두 번째 프로세스는 즉시 거부됩니다. stale 락(기록된 PID가 죽음 — os.kill(pid,0)로 확인)은 자동 삭제하지 않고 알리기만 합니다 — 진짜 실행 중인 디스패치를 죽이지 않으려고. 락 해제는 3중(__exit__ · atexit · SIGINT/SIGTERM)이라 비정상 종료에도 정리됩니다. fcntl 없는 플랫폼이면 락을 건너뛰고 경고만(단일 사용자 가정).

스냅샷 — diff 기준선은 SHA 하나

왜 stash가 아니라 SHA인가

구현 직전 엔진은 git rev-parse HEAD — SHA 문자열 하나 — 를 이 디스패치의 diff 기준선으로 박습니다. 게이트(scope guard · contract sync · TDD first-commit)는 "이 SHA 이후 무엇이 바뀌었나"를 봅니다. working tree는 전혀 건드리지 않습니다: stash도, worktree 교체도 없습니다. 캡처가 실패하면 디스패치를 아예 시작하지 않습니다 — 기준선 없이 게이트를 돌리지 않습니다.

게이트 실패 → 정확히 1회 리트라이

게이트 실패가 곧바로 block은 아닙니다. 엔진은 실패를 post-fail hook(stderr 패턴)과 매칭하고, 매칭되면 hook을 실행한 뒤 hook.retry가 켜진 경우에만 게이트를 1회 재실행합니다. 통과 → done, 여전히 실패 → blocked. 무한 루프가 아니라 정확히 1회 — "알려진 일시적 실패"(예: 환경 셋업 누락) 교정용입니다.

self-block short-circuit: 구현가가 핸드오프에서 Block: Q-XXX를 반환하면 엔진은 즉시 blocked 처리하고 게이트를 건너뜁니다 — 헛된 리뷰 비용 없음. CODEX 쿼터 소진은 subprocess stderr에서 감지해 기록 → 소진된 쿼터에 헛된 재시도를 던지지 않습니다.

13SSOT 동시성 — 파일 락 · 원자적 쓰기

QUEUE/ARCHIVE는 평범한 YAML이라 동시 쓰기 = 손상 위험이 있습니다. 상태 레이어가 모든 쓰기를 advisory 파일 락 + 원자적 교체로 막습니다.

  • 파일 락 — .<name>.lock를 fcntl.flock(LOCK_EX|LOCK_NB)(배타·논블로킹)로. 경합 시 0.1s 폴링, 30s 타임아웃이면 hang 대신 에러. fcntl 없는 플랫폼용 명시적 env 탈출구 존재(경고: race 가능).
  • 원자적 쓰기 — 모든 쓰기는 임시 파일에 먼저 쓰고 os.replace()(POSIX 원자적 rename) — 반쪽짜리 파일을 읽는 일이 없습니다.

세 개의 분리된 락 — 디스패치와 archive 공존

락 파일보호 대상핵심
.QUEUE.yaml.lockQUEUE 쓰기상태 전이 시 배타
.archive.lock디스패처 sentinel"이관 진행 중" 표시 → archive 도중 dispatch 거부
.archive-index.lockarchive-index 재빌드sentinel과 별개라 재인덱싱이 dispatch를 막지 않음

분리는 의도된 설계: archive 이관은 dispatch를 멈춰야 하지만 인덱스 재빌드는 그럴 필요가 없습니다. 그 인덱스가 os3 lookup이 archive된 티켓을 찾게 해주고, 인덱스도 원자적으로 재빌드됩니다.

14세션 경계의 기억

AI 세션은 컨텍스트가 날아갑니다. OS는 시작/종료 프로토콜로 기억을 파일에 고정해, load-bearing한 것이 채팅창에만 살지 않게 합니다.

Boot read map — 한 번만 읽기

세션 시작은 각 SSOT 파일을 1회만 읽고 머릿속에 요약을 유지합니다. 같은 파일을 (offset 바꿔) 다시 읽으면 캐싱 실패 신호로 봅니다. sub-agent는 토큰 절약을 위해 슬림 코어로 부팅하고, 전문 헌법은 필요할 때만 읽습니다.

세션 종료 — 8축 회고

종료는 "오늘 한 일"이 아닙니다. 8개 카테고리를 의무적으로 자문합니다 — 없어도 "없음"이라고 답해야 합니다:

#자가질문
1user↔AI 협업 마찰 (같은 지시 반복? 의도 오해?)
2DB/data 위험 (schema drift, 마이그레이션 누락, 데이터 손실)
3dev 환경 (token TTL, redirect loop, port 충돌, env 누락)
4잔재 파일 / .gitignore (임시파일, sentinel, working-tree 잔재)
5분기 / 미뤄둔 PM 결정 (미해결 결정, unblock 누락)
6기억 — 이번에 본 패턴이 장기 memory 자격인가?
7vendor 정보 — 가정하지 않고 --help로 검증했나?
8보안 — 평문 secret, API key, OAuth flow, OWASP 신호

회고 헤더엔 revision_delta(첫 초안이 빠뜨린 항목 수 vs 최종)를 기록합니다. 게이트가 아니라 메트릭 — 자기 누락률을 스스로 보게 만드는 압력.

두 갈래의 기억

프로젝트 상태(공유·git 추적)는 devos/에, 사용자/협업 선호(개인·git 밖)는 harness의 memory에 갑니다. 규칙은 같은 사실을 양쪽에 중복하지 않는 것. 별도의 append-only learnings.jsonl은 재사용 가능한 인사이트를 로깅합니다. 장기 memory는 트리거에서만 — 사용자 교정, 명시된 선호, 검증된 라이브러리 quirk, ≥2회 반복 패턴 — 코드/git에서 쉽게 재발견되는 건 안 적습니다.

15문제 관리 — 결정 · 막힘 · 인바운드 이슈

문제는 종류별로 다른 채널에 들어갑니다. 채팅에서 "어 이거 문제네"로 끝나지 않고 파일에 박제됩니다.

① Question Queue (Q-XXX) — 결정이 필요한 것

막힌 티켓이나 아키텍처 갈림길은 questions/QUEUE.md에 표준 형식으로 적습니다. 핵심은 Options + Recommendation + Default — 무응답이어도 Default로 진행 가능하게.

## Q-0XX: <짧은 제목>
Status: [open] | [resolved] | [parked]
Blocking: yes | no
**Options**: A) … B) … C) …
**Recommendation**: A + B — <이유>
**Default (if no response)**: A
**Resolution (date, actor)**: …

세션 시작 시 계획가가 모든 [open]을 모아 triage합니다(Blocking 우선, 상한 있음; 오늘 티켓에 영향 없는 non-blocking은 묻지 않고 Default 가정).

② blocked — 진행이 멈춘 것

티켓 자체에 _blocked_reason + _blocked_log + 전이 이력으로 기록(§3). 막힘은 보통 엔진 자동 정지(preflight·deps·게이트 실패)거나 검토 BLOCKER이며, 둘 다 대개 Q-XXX로 사용자 결정을 요청합니다.

③ 인바운드 이슈 (I- / P- / A-)

같은 호스트를 쓰는 sister 프로젝트가 발견한 OS 차원 결함을 인계합니다: I = 버그, P = 제안/DX 개선, A = 아키텍처/감사. 처리: cross-reference(이미 고쳐졌나) → 다축 분석 → 같은 근본원인 항목 묶기 → 몇 개 결정 포인트로 추상화 → plan.

즉흥 수정이 아니라 추상화

N건의 보고는 한 줌의 메커니즘 fix가 되지 N티켓이 되지 않습니다 — 같은 근본원인 항목은 묶입니다. 그리고 처리 자체가 반복되면 그게 다시 OS 메커니즘이 됩니다(인바운드 템플릿이 그런 산물).

16정책 심화 — 웨이버 · hard/soft · 역할 권한

Waiver (W-XXX) — 예외는 기록되어야 통과한다

Production hard policy 예외는 "그냥 넘어가기"가 안 됩니다. inline 웨이버 레코드 + PM 승인이 있어야 합니다. 없으면 작업은 done이 아니라 needs_pm/blocked에 머뭅니다.

waivers:
  - id: W-YYYYMMDD-NNN
    policy: required_visual_review | security_review | test_failure | coverage | reviewer_request_changes
    requested_by: CODEX
    approved_by: PM            # 반드시 PM
    decision: accept_with_waiver
    reason: <위험을 이해하고 받아들인 이유>
    risk_accepted: <구체적 잔여 위험>
    expires: never | YYYY-MM-DD | after_ticket
    follow_up_ticket: T-YYY | none

웨이버 가능 (PM 임시 통과)

  • non-critical 테스트 실패 (follow-up 존재)
  • 주관적 visual / 취향 트레이드오프
  • coverage 부족 (위험 수용 + follow-up)

웨이버 불가 (절대)

  • secret 노출 · owner mismatch · scope 위반
  • 미해결 deps · dirty-tree destructive 위험
  • remediation 없는 auth/payment/privacy/data-loss

Hard vs Soft 정책 클래스

Hard = Production에서 fail-closed: 중복 id, 미해결 deps, owner mismatch, scope 위반, secret, required 테스트 실패, required 리뷰 reject, UI visual 누락, 웨이버 누락. Soft = 경고 / cleanup / follow-up 티켓: 파일 길이 budget, 복잡도, 문서 drift, 스타일, 선택적 리팩터.

역할 · 권한 매트릭스

도구 권한(§7)은 에이전트별이고, 그 위에 역할별 의사결정 경계가 있습니다. 한 에이전트가 여러 역할을 겸할 수 있습니다.

17측정 — 비용 · 점수 카드 · paired-run

비용 채널

os3 cost-report가 디스패치 로그를 세 채널로 집계해 비용을 추정합니다: C1 interactive(reviewer / security / designer), C2 agent-SDK(더 싼 배치 리뷰), C0 외부 CODEX(구현 + b' 2차 의견). 한 채널이 크레딧의 ~80%를 넘으면 경고 — "어느 모델에 얼마나 리뷰를 시켰나"를 돈으로 환산.

4축 점수 카드 (격주)

Phase ship 직후 + 격주로 4축을 기록해 균형을 감시합니다. 각 축의 ship/위험 임계:

축핵심 임계
Qualityfindings_recall ≥ 90% · blocker_misses = 0 · mutation 생존자 ≤ 5% · user_acceptance ≥ 80%
Speed티켓 wallclock · baseline 대비 delta(음수=빨라짐)
CostC1/C2/C0 추정 + 합계 · baseline 대비 delta(음수=절감)
Securitycritical 오분류 = 0 · b' escalation 10–25% · critical full-chain = 100%
vetokill — 즉시 제동

다음 중 하나라도 참이면 자동 제동: blocker_miss ≥ 1(rollback) · critical 오분류 ≥ 1 · 속도 −20% · cost 증가. verdict는 continue / pause / rollback — 최적화 전체에 대한 계측된 회로차단기.

paired-run — 모델 교체 전 실측

구현가를 더 싼 모델로 옮기기 전에, 같은 티켓을 두 모델로 돌려 품질+비용을 실측합니다. default 모델은 임계 충족 시에만 바뀝니다(예: ≥3티켓 · recall ≥ 90% · BLOCKER 누락 0 · acceptance 유지). 즉 "싼 모델로 갈아타기"가 감이 아니라 실측으로 일어납니다.

토큰 규율이 함께 갑니다: QUEUE↔ARCHIVE 분리로 활성 파일을 작게, sub-agent는 슬림 코어로 부팅, 디스패치가 그 코어를 prompt에 인라인해 read 라운드트립 생략, context-monitor 훅이 잔량 ~35% / ~25%에서 경고해 thrashing 전에 핸드오프합니다.
