How the OS actually works

A deep dive into the layers, agents, and enforcement of Vibe Coding OS — the part the README summarizes. Written for people who want to understand why it's safe, not just that it is.

0One paragraph

Vibe Coding OS turns "telling an AI to write code" into an auditable state machine. All truth lives in files under devos/; work moves only as tickets; and who did what, and why, is recorded in per-ticket transition history and session logs.

You (the PM) describe intent → CLAUDE1 decomposes it into tickets and interrogates the missing error/edge/security cases → you approve the plan → the ticket's owner (builder in-session or CODEX subprocess) implements → baseline gates + a read-only review board check it → if risky or uncertain, a different vendor (CODEX b') gives a second opinion → no BLOCKER means done; a BLOCKER means blocked + a decision queued for you.

1Three layers

The system is three layers. The top two are "what it does"; the bottom one is "what stops you breaking the rules." Most of the safety comes from the third.

LayerWhatRole
1 · Statefile-based SSOT (devos/)remembers what the project is and where it stands
2 · Agentsplanner + 2 implementers + read-only review boardplan, implement, review
3 · Enforcementtool permissions · hooks · gates · grep · locked decisionsmakes the rules structural, not trust-based

Core principle: roles and permissions are separated, and permission is limited at the tool layer. A reviewer has no Edit/Write tool — it literally cannot "fix" what it found, so self-review objectivity can't collapse.

2The agents

Each agent uses a different model and a different set of tools. Permission separation is enforced by structure, not goodwill.

CLAUDE1 (main)Opusplan · orchestrate

Talks to you. Decomposes PRDs into tickets, researches, routes work. Does not write production code; does not OK results directly — that goes to the review board.

builderSonnetRead·Edit·Write·Bash

In-session implementer. Preferred for ambiguous, experience-heavy UI and new UX flows. Touches only the ticket's declared files.

CODEXexternal CLIsubprocess · sandbox

Platform implementer — backend/API/data/infra/tests/migrations — and the cross-model second opinion (b'). A different vendor, so it catches blind spots Claude shares.

reviewer · designer · securityREAD-ONLY

The review board (Opus/Sonnet). reviewer = adversarial PR review (DOD↔test, assertions, scope, contracts). designer = UI/UX (consistency, states, accessibility). security = OWASP + STRIDE, auto-invoked for auth/payment/permissions/external input. None has an Edit/Write tool.

Why read-only matters

Most teams say "don't merge your own PR" as policy. Here it's permission: the reviewers have no write tool, so they report findings and cannot quietly fix them. Structural objectivity.

3Tickets — the unit of work

Every implementation is a ticket in devos/tasks/QUEUE.yaml. 1 ticket = 1 PR. A ticket is self-contained — an implementer reading only the ticket + SSOT can execute it.

- id: T-AUTH-LOGIN-01
  status: todo            # the dispatcher only picks up `todo`
  owner: CODEX            # BUILDER | CODEX
  mode: production        # gate posture (see §6)
  tdd: required           # required | skip | self-evident
  security_audit: true    # auto-forced for auth/payment/permissions/...
  goal: |   POST /auth/login
  dod:
    - 'valid credentials -> 200 + access/refresh JWT'      # success
    - 'wrong password -> 401 + "invalid credentials"'      # error (mandatory)
  files: [apps/api/src/auth/login.py, tests/test_login.py]
  verify: |  python3 -m pytest tests/test_login.py -v
  gates: [pr-check]

DOD rule: if a success-case DOD exists, a failure/error-case DOD is mandatory. This is how the system fills the "non-developer writes only the happy path" gap.

Status machine + audit trail

States: todo doing code_ready needs_pm done blocked parked. Every transition records actor · reason · timestamp into the ticket's _transition_history — so you reconstruct why a ticket is where it is from the file, not the chat.

Every dispatch reports a standard header — which path, which model built it, who reviewed it, the verdict — so you never have to ask.

5Dispatch (deterministic)

When CLAUDE1 dispatches a ticket it follows a fixed 7-step procedure:

StepDoes
1 · lookup + preflightstatus==todo? deps all done? declared files exist?
2 · owner routingBUILDER → in-session sub-agent · CODEX → subprocess
3 · implementimplementer does the work in its own context
4 · pr-check gatesecrets · contract sync · scope guard · session log · TDD first-commit
5 · review chainclassify ticket → spawn the needed read-only reviewers (+ a 5% random sample escalates light paths)
6 · fan-in + b'collect verdicts; on any quantitative trigger, call a CODEX second opinion
7 · finalizeBLOCKER-wins verdict → status + log + standard result header
Deterministic engine, disciplined wrapper

The engine (dispatcher.py) is deterministic in code — singleton lock, an HEAD-SHA diff baseline (no git stash), gates, a single retry. The orchestration that wraps it (classification, the b' trigger) is executed by CLAUDE1 to the protocol above.

6Operating modes

One idea moves exploration → productization → production as confidence rises. The mode sets the gate posture, enforced by the dispatcher.

Gateexplorationproductizationproduction
Testsoptional/report-onlystrategy requiredrequired
Reviewerreport-onlyrecommendedrequired · reject blocks
Securitysecrets/auth onlyrisk IDrequired when triggered
Success+failure DODoptionalrequiredrequired

Report-only is fail-closed: a gate is softened only when the mode is exploration/productization and the gate is an explicitly recognized soft gate. Secrets, file-scope violations, destructive actions, and unknown gates block in every mode. Production exceptions need a recorded waiver (who approved, what risk, follow-up).

7Safety — structure, not trust

🔒 Permission-enforced objectivity

Reviewers have read-only tools. The planner is blocked by a hook from writing to apps/ packages/ scripts/ tests/. Both are tool-layer facts, not promises.

⚖️ Iron Laws (all agents)

No prod code without a failing test first · no fix without root-cause · no "done" without fresh evidence · no merge without adversarial review · no scope-reduction vocabulary (grep-enforced).

🚪 Always-on safety floor

Secret exposure, owner mismatch, file-scope violation, unresolved deps, destructive-with-dirty-tree block regardless of mode and can't be waived normally.

🤝 Cross-model net (b')

When the reviewer is uncertain, a BLOCKER appears, or the ticket is high-risk, a different vendor gives a second opinion. Free when calm.

Incident → Locked Decision

When something breaks, the fix becomes an enforced rule — a D-XX Locked Decision whose violation is an automatic reviewer BLOCKER — so it can't recur. The rulebook is a record of past incidents, not a wishlist.

8How LLMs read the OS (SSOT)

All state is files. Agents reload them at session start, so there's no shared chat window to drift. When sources disagree, higher wins:

1. PROJECT_STATE.md        milestones, what works now
2. API_CONTRACT / UI_CONTRACT   interface contracts (above code)
3. ADR/*                   architecture decisions
4. tasks/QUEUE.yaml        active tickets
5. Code
6. session logs
7. chat logs               (least reliable)

Contract-first: if API/UI behavior changes, the contract doc is updated before the code; a reviewer flags a mismatch as a BLOCKER.

9The host-OS model

The engine, agents, and doctrine live once in a host repo. Each product is an independent git repo under projects/, carrying only its own code and task state.

~/dev-os/            host = engine · doctrine · shared .claude (one git repo)
|-- server/  bin/os3  .claude/  devos/
`-- projects/        (git-ignored by host — each is its own repo)
    |-- product-a/   apps/ packages/ + own devos/tasks · PROJECT_STATE · .os3.yaml
    `-- product-b/   ...
  • Engine in one place — improve it once; every project gets it. No sync.
  • Projects independent — own repo, own state, deployed separately.
  • One entry — os3 open <name> enters any project with host settings attached.

Using it as a single project works too — keep devos/ as your state and ignore projects/. The host-OS model is how it scales to several products.

10The OS-feedback loop

The OS improves itself demand-pulled: friction hit during product work is captured with os3 feedback "..." into devos/os-feedback/INBOX.md, reviewed at session start, and converted to tickets during a periodic consolidation pass.

Why it's load-bearing

If OS improvement is supposed to come "from product friction," that friction has to be captured — otherwise it's mentioned once in chat and evaporates. Reactive fixing catches what you hit; a scheduled consolidation pass catches the latent debt you don't trip over.

12Engine internals — lock · snapshot · retry

§5 was "what steps the dispatch runs." This is how the engine runs them safely. Three pieces: no double-run · a fixed change-baseline · one retry on failure. Note what's absent — there is no git-mutating rollback; recovery is read-only by design.

Singleton lock — no two dispatchers per project

A dispatch acquires a lock by atomic file creation (O_CREAT|O_EXCL) at ~/.os3/dispatcher.<project>.pid. A second process is refused immediately. A stale lock (the recorded PID is dead, checked via os.kill(pid,0)) is reported, not auto-deleted — so a genuinely running dispatch is never killed. The lock is released three ways (__exit__ · atexit · SIGINT/SIGTERM) so abnormal exits still clean up. On a platform without fcntl it skips the lock and warns (single-user assumption).

Snapshot — the diff baseline is one SHA

Why a SHA, not git stash

Just before implementation the engine records git rev-parse HEAD — one SHA string — as this dispatch's diff baseline. The gates (scope guard, contract sync, TDD first-commit) ask "what changed since this SHA." The working tree is never touched: no stash, no worktree swap. If the capture fails, the dispatch does not start — gates never run without a baseline.

Gate fail → exactly one retry

A failing gate isn't an instant block. The engine matches the failure against a post-fail hook (by stderr pattern); if one matches it runs the hook, and only if hook.retry is set does it re-run the gate once. Pass → done; still failing → blocked. It is exactly one retry, not a loop — for known transient failures (e.g. missing env setup).

Self-block short-circuit: if an implementer returns Block: Q-XXX in its handoff, the engine marks blocked immediately and skips the gates — no wasted review spend. CODEX quota exhaustion is detected from subprocess stderr and recorded so retries aren't thrown at an exhausted quota.

13SSOT concurrency — file locks · atomic writes

QUEUE/ARCHIVE are plain YAML, so concurrent writes could corrupt them. The state layer guards every write with an advisory file lock plus an atomic replace.

  • File lock — .<name>.lock via fcntl.flock(LOCK_EX|LOCK_NB) (exclusive, non-blocking). On contention it polls at 0.1s; a 30s timeout raises an error rather than hanging. An explicit env opt-out exists for platforms without fcntl (warns: race possible).
  • Atomic write — every write goes to a temp file then os.replace() (POSIX atomic rename), so no one ever reads a half-written file.

Three separate locks — dispatch and archive coexist

Lock fileProtectsPoint
.QUEUE.yaml.lockQUEUE writesexclusive during a state transition
.archive.lockdispatcher sentinelsignals "archiving in progress" → dispatch is refused mid-archive
.archive-index.lockarchive-index rebuildseparate from the sentinel, so re-indexing doesn't block dispatch

The split is intentional: an archive move must pause dispatch, but an index rebuild need not. That index is what lets os3 lookup find archived tickets — and it too is rebuilt atomically.

14Memory across sessions

An AI session loses its context. The OS pins memory to files with a start/end protocol, so nothing load-bearing lives only in a chat window.

Boot read map — read once

Session start reads each SSOT file once and keeps a summary in head; re-reading the same file (different offset) is treated as a caching-failure signal. Sub-agents boot from a slim core to save tokens and read the full constitution only on demand.

Session end — an 8-axis retro

The close isn't "what I did today." It's a mandatory self-interrogation across 8 categories — answer each, even if "none":

#Self-question
1user↔AI collaboration friction (same instruction repeated? intent misread?)
2DB/data risk (schema drift, missed migration, data loss)
3dev environment (token TTL, redirect loop, port clash, missing env)
4leftover files / .gitignore (temp files, sentinels, working-tree residue)
5branching / deferred PM decisions (unresolved choices, missed unblock)
6memory — is a pattern seen this session worth long-term memory?
7vendor claims — did I verify with --help rather than assume?
8security — plaintext secret, API key, OAuth flow, OWASP signal

The retro header records a revision_delta (how many items the first draft missed vs the final). It's a metric, not a gate — pressure to see your own omission rate.

Two memory tracks

Project state (shared, git-tracked) lives in devos/; user/collaboration preferences (personal, outside git) live in the harness's memory. The rule is not to duplicate the same fact across both. A separate append-only learnings.jsonl logs reusable insights. Long-term memory is written only on triggers — a user correction, a stated preference, a verified library quirk, or a pattern seen ≥2× — never for things easily re-derived from code or git.

15Problem management — decisions · blocks · inbound issues

Problems go into different channels by kind. None of them end as "huh, that's a problem" in chat — they're pinned to files.

① Question Queue (Q-XXX) — something needs deciding

A blocked ticket or an architecture fork is written to questions/QUEUE.md in a standard shape. The key is Options + Recommendation + Default — so the system can proceed on the Default even if you never answer.

## Q-0XX: <short title>
Status: [open] | [resolved] | [parked]
Blocking: yes | no
**Options**: A) … B) … C) …
**Recommendation**: A + B — <why>
**Default (if no response)**: A
**Resolution (date, actor)**: …

At session start the planner gathers every [open] and triages (blocking first, capped; a non-blocking item that doesn't touch today's tickets is assumed at its Default rather than asked).

② blocked — progress stopped

Recorded on the ticket itself as _blocked_reason + _blocked_log + the transition history (§3). A block is usually an engine auto-stop (preflight, unresolved deps, gate fail) or a review BLOCKER — both typically raise a Q-XXX for you.

③ Inbound issues (I- / P- / A-)

A sister project using the same host can hand off OS-level defects it hit: I = bug, P = proposal/DX improvement, A = architecture/audit. Handling: cross-reference (already fixed?) → multi-axis analysis → cluster items sharing a root cause → abstract to a few decision points → plan.

Abstracted, not impulse-fixed

A deck of N reported items becomes a handful of mechanism fixes, not N tickets — items with the same root cause are merged. And when the handling itself recurs, that becomes its own OS mechanism (the inbound template is one such product).

16Policy depth — waivers · hard/soft · role authority

Waiver (W-XXX) — an exception must be recorded to pass

A production hard-policy exception can't "just slide." It needs an inline waiver record + PM approval; without one, the work stays needs_pm/blocked rather than done.

waivers:
  - id: W-YYYYMMDD-NNN
    policy: required_visual_review | security_review | test_failure | coverage | reviewer_request_changes
    requested_by: CODEX
    approved_by: PM            # must be PM
    decision: accept_with_waiver
    reason: <risk understood and accepted>
    risk_accepted: <specific residual risk>
    expires: never | YYYY-MM-DD | after_ticket
    follow_up_ticket: T-YYY | none

Waivable (PM may pass)

  • non-critical test fail (follow-up exists)
  • subjective visual / taste trade-off
  • coverage gap (risk accepted + follow-up)

Never waivable

  • secret exposure · owner mismatch · scope violation
  • unresolved deps · dirty-tree destructive risk
  • unremediated auth/payment/privacy/data-loss

Hard vs soft policy class

Hard = fail-closed in production: duplicate id, unresolved deps, owner mismatch, scope violation, secret, required-test fail, required-review reject, missing UI visual review, missing waiver. Soft = warning / cleanup / follow-up ticket: file-length budget, complexity, doc drift, style, optional refactor.

Principle: the PM doesn't parse raw policy noise — the system summarizes what blocks, what's a recommendation, and what needs PM judgment, and routes only the last to you.

17Measurement — cost · scorecard · paired-run

Cost channels

os3 cost-report aggregates the dispatch logs into three channels and estimates spend: C1 interactive (reviewer / security / designer), C2 agent-SDK (cheaper batch review), C0 external CODEX (implement + the b' second opinion). It warns when a channel passes ~80% of its credit — "how much review did I put on which model" turned into money.

4-axis scorecard (biweekly)

Right after a phase ships, and on a biweekly cadence, four axes are recorded to watch the balance, each with a ship/risk threshold:

AxisKey thresholds
Qualityfindings_recall ≥ 90% · blocker_misses = 0 · mutation survivors ≤ 5% · user_acceptance ≥ 80%
Speedticket wall-clock · delta vs baseline (negative = faster)
CostC1/C2/C0 estimate + total · delta vs baseline (negative = saved)
Securitycritical misclassification = 0 · b' escalation 10–25% · critical full-chain = 100%
vetokill — instant brake

Any one of these auto-brakes: blocker_miss ≥ 1 (rollback) · critical misclassification ≥ 1 · speed −20% · cost up. The verdict is continue / pause / rollback — a measured circuit-breaker on the whole optimization.

paired-run — measure before switching models

Before moving an implementer to a cheaper model, the same ticket is run on both models and quality+cost are measured. The default model changes only after a threshold is met (e.g. ≥3 tickets · recall ≥ 90% · 0 BLOCKER misses · acceptance held). So "switch to the cheaper model" happens on measured evidence, not vibes.

Token discipline rides alongside: the QUEUE↔ARCHIVE split keeps the active file small, sub-agents boot from a slim core, dispatch inlines that core into the prompt to skip a read round-trip, and a context-monitor hook warns at ~35% / ~25% remaining so a session hands off before it thrashes.
